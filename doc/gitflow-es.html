GitFlow

El flujo de trabajo de git más famoso 

git flow init

Initialized empty Git repository in ~/project/.git/
No branches exist yet. Base branches must be created now.
Branch name for production releases: [master]
Branch name for "next release" development: [develop]

How to name your supporting branch prefixes?
Feature branches? [feature/]
Release branches? [release/]
Hotfix branches? [hotfix/]
Support branches? [support/]
Version tag prefix? []

Solo 2 ramas fijas

  • Develop (Rama de integración para features):
  • Master (Todos commits están tagged con versión)

3 canales de ramas temporales

  • Feature branches: Develop –> Develop
  • Release branches: Develop –> Master/Develop
  • Hotfix branches: Master –> Master/Develop

Ramas Features

  • De Develop a Develop
  • Ningun interacción con Master
  • Creado a partir del último commit en Develop

Pasos

Crear con: 

git checkout develop
git checkout -b feature_branch   

git flow feature start feature_branch

Cuando acabes: 

git checkout develop
git merge feature_branch

git flow feature finish NAME

Release

  • Fork de Develop
  • Nada nuevo, solo bugfix, documentación, …
  • Refinar Release mientras de trabajar en nuevas Features en Develop
  • Cuando esté listo
    • Merge a Master con version como tag
    • Merge a Develop para asegurar sincronización
  • Oportunidad de revisar el código por otras personas y crear pull requests

Pasos

Crear con: 

git checkout develop
git checkout -b release/0.1.0

git flow release start 0.1.0

Cuando acabes: 

git checkout master
git merge release/0.1.0

git flow release finish '0.1.0'

Hotfix

  • Se crea solo desde Master
  • Se hace un merge a Master y Develop
  • El commit en Master necesita su propia versión

Pasos

Crear con: 

git checkout master
git checkout -b hotfix_branch

git flow hotfix start hotfix_branch

Cuando acabes: 

git checkout master
git merge hotfix_branch
git checkout develop
git merge hotfix_branch
git branch -D hotfix_branch

git flow hotfix finish hotfix_branch

Versioning

MAJOR.MINOR.PATCH:

  1. MAJOR cambios incompatible con versiones anteriores
  2. MINOR functionalidad compatible con versiones anteriores
  3. PATCH correcciones de errores compatibles con versiones anteriores

Ejemplo:

  • 0.1.0 primera version
  • 0.2.0 anadido functionalidad
  • 0.2.1 correccion de error
  • 1.2.1 cambio de technologia (e.g. nueva plataforma)

Summary

  1. Develop creado a partir de Master
  2. Ramas de Features se crean a partir de Develop
  3. Cuando una Feature está completa, se fusiona en la rama de Develop
  4. Ramas de Release se crean a partir de Develop
  5. Cuando una Release está completa, se fusiona se fusiona a Master y Develop
  6. Se crea una rama Hotfix cuando se detecta un error en Master
  7. Cuando una Hotfix está completa, se fusiona se fusiona a Master y Develop

En concreto

  1. Cada historia en una unica rama Feature compartida
  2. Historia completa solo cuando completa/documentada/probada
  3. Cada Release lleva un changelog y esta approbada por todo el equipo
  4. Entorno Dev esta en rama Develop
  5. Entorno Pre esta basada en ultima rama Release
  6. Entorno Pro esta en rama Master
